# Proposal Outline

## 开场白

开始执行时，先向用户说明：

- 本 skill 用于公司内部立项的结构化搭建
- 默认项目运行在微信小程序多人联机游戏平台中
- 本阶段会把已有想法映射为标准目录，并找出最影响立项判断的缺口
- 会继续沿用一次一个问题、进度提示和 `A / B / C` 选项的交互方式

## 目标

把 `00_project_brief.md` 变成可直接进入全文写作的标准大纲。

## 文档边界

本阶段输出的是立项文档大纲，不是完整策划案大纲。

- 章节重点是支撑立项判断
- 系统设计写到核心结构即可
- 关卡、角色、数值只需要覆盖 MVP 必需范围
- 如果原型有 4x4 关卡，立项阶段通常只需设计 1-2 个代表性关卡

## 展开重点

立项文档中，核心玩法相关内容需要比商业化相关内容展开得更充分。

- 操作方式
- 核心循环
- 地图和空间
- 敌人规则
- 角色设计
- build / 商店 / 强化结构
- 乐趣点与验证方式

商业化、长线运营、完整系统表可以保留方向性描述，不需要在立项阶段一次写全。

## 交互要求

- 每次追问只补 1 个最影响成稿质量的缺口
- 选择题尽量写成 `A / B / C`
- 阶段推进时提供粗粒度进度提示

## 输入

- `00_project_brief.md`
- 如有需要，可参考 `references/document-structure.md`
- 如有旧项目文档，可提炼其优势表达方式，但不能照抄

## 执行方式

1. 按标准模板建立一级目录。
2. 每个一级章节下，补充：
   - 已有信息
   - 需要补充
   - 建议写法
3. 对信息不足的章节，生成一个精简问题清单。
4. 优先识别最影响立项判断的缺口，例如：
   - 市场机会是否成立
   - 核心玩法是否真的有差异化
   - 多人联机是否真的创造了额外价值
   - 竞品是否已经被验证、是否已有移动端版本
   - 商业化和留存是否能在公司平台成立
   - 成长与商业化是否冲突
   - MVP 是否可验证

## 输出要求

必须输出两份文档。

### `01_outline.md`

```md
# 策划案大纲

## 一、文档基础信息
- 已有信息：
- 需要补充：
- 建议写法：

## 二、项目概述
...
```

### `01_open_questions.md`

```md
# 待补问题清单

## 高优先级
1.
2.
3.

## 中优先级
1.
2.

## 可后置
1.
2.
```

## 章节判断原则

- 如果缺少事实，不要硬写成结论
- 如果章节价值不高，可以建议压缩写法
- 如果旧稿里有更强的推演逻辑，优先保留
- 如果“多人”只停留在人数层面，要明确标成风险

## 完成标准

- 所有一级章节都已占位
- 已能看出每章打算写什么
- 待补问题数量可控，且按优先级排序
