# Proposal Intake

## 开场白

开始执行时，先用 1-2 句中文向用户说明：

- 本 skill 用于公司内部立项文档梳理
- 默认场景是微信小程序多人联机游戏平台
- 会一次只问一个关键问题，逐步收束到可写的项目简报
- 会给出粗粒度进度提示

## 语言要求

- 不要使用明显性别指代，例如“他”“她”
- 统一使用“用户”“玩家”“队友”“对手”等中性称谓
- 提问和归纳时避免带偏见的默认假设

## 目标

把一个模糊想法收束成可进入写作阶段的项目简报。

## 固定业务前提

除非用户明确推翻，否则默认以下条件成立：

- 平台是微信小程序
- 项目属于多人联机游戏平台
- 游戏需要支持多人游玩

因此不要把“是否多人”“是否微信小程序”当作常规首轮问题，优先追问真正影响立项方向的差异点。

## 项目分类

先判断当前项目属于以下哪一类，再决定提问策略：

### A. 有多人联机原型游戏

例如用户说“做小程序版《枪豆人》”。

此类项目重点关注：

- 原型游戏是否已经验证过好玩
- 原型的市场表现、口碑、数据是否值得参考
- 是否已有官方移动端版本
- 如果已有移动端版本，本项目还要不要做差异化
- 放到公司平台后，项目竞争力是什么
- 免费游玩 + IAP 持续运营模式如何成立
- 留存和付费内容可能落在哪里

### B. 有单机原型游戏

例如用户说“把《潜水员戴夫》改成多人联机”。

此类项目重点关注：

- 单机核心体验是否适合多人化
- 多人改造是否会损伤原有乐趣
- 多人游玩方式如何成立
- 联机改造后的节奏、目标和反馈是否仍然顺畅
- 平台适配与商业化如何补足

### C. 没有成熟原型，或只有少量参考

此类项目重点关注：

- 玩法本身是否有足够吸引力
- 多人联机价值是否真实存在
- 是否有可借鉴的验证对象
- 商业化、留存、MVP 验证是否能闭环

## 类型化追问

除项目来源分类外，还要判断游戏类型，并主动追问该类型最关键的立项内容。

### 动作 / 射击 / 肉鸽 / 闯关类

优先追问：

- 基础操作方式是点按、摇杆，还是其他输入
- 玩家如何移动、输出、躲避
- 地图大小与空间压力
- 敌人出现方式与刷新节奏
- 地图边界、失败条件与复活规则
- 角色设计和差异
- 商店、build 和强化结构

### 模拟经营类

优先追问：

- 模拟对象是什么
- 经济循环怎么跑
- 用户长期目标是什么
- 经营和操作的节奏如何切换
- 包装和故事是否承担驱动力

### 其他类型

优先追问能够支撑该类型成立的核心决策、操作、反馈与验证方式。

## 执行方式

1. 先判断用户当前信息成熟度：
   - 只有一个点子
   - 已有一些玩法或竞品参考
   - 已经有较完整想法但缺少结构化表达
2. 每轮只问 1 个最关键问题。
3. 先问会影响整篇文档方向的问题，不要一上来问细枝末节。
4. 如果用户答案不完整，先帮用户做归纳，再继续追问。
5. 未知信息必须标记为 `[待确认]` 或 `[假设]`。
6. 尽早确认“多人联机”的具体价值，而不是默认它天然成立。
7. 如果用户已经明确给出强参考竞品，不要再追问过于宽泛的体验问题，应优先问“你要继承什么、改什么、删什么”。
8. 如果是多人原型对标项目，优先问竞品验证、移动端情况、平台差异化和商业化，不要过早陷入技术细节。
9. 如果是单机改多人项目，优先问多人改造是否破坏原体验。
10. 如果是创新项目，优先问玩法吸引力和 MVP 验证方式。
11. 如果用户已明确给出竞品名称，先做基础检索，再继续提问。
12. 给出选择题时，尽量使用 `A / B / C` 格式。
13. 如果是动作 / 射击 / 肉鸽类项目，商业化可以先收着问，但操作、地图、敌人、边界、角色、商店 build 必须问清楚。

## 必采字段

- 项目名称或临时代号
- 一句话定位
- 项目目标
- 目标用户
- 核心卖点
- 多人联机价值
- 参考竞品
- 当前最大不确定项

## 推荐追问顺序

### 第一轮

- 你想做的到底是什么类型的项目
- 这个项目里，多人一起玩带来的核心体验是什么
- 玩家最先感受到的核心体验是什么

如果用户已经明确说出“想做某个已知竞品的小程序版 / 轻量版 / 联机版”，则改用以下顺序：

0. 先搜索并确认竞品的基础玩法、平台信息和公开验证信号
1. 先确认准备继承的核心流程是什么
2. 再确认准备改造的差异点是什么
3. 再确认多人联机如何放大这个差异点

如果判断为 A 类项目，再优先补问以下内容：

1. 这个原型游戏为什么值得抄，验证证据是什么
2. 它有没有移动端版本，如果有，本项目差异化在哪里
3. 放到公司平台后，付费和留存准备怎么做

如果判断为 B 类项目，再优先补问以下内容：

1. 单机原作最核心的体验是什么
2. 多人改造会不会伤害这个体验
3. 多人具体怎么一起玩才成立

如果判断为 C 类项目，再优先补问以下内容：

1. 核心玩法为什么会吸引用户
2. 多人价值怎么成立
3. MVP 最小验证闭环是什么

### 第二轮

- 目标用户是谁
- 这件事为什么值得做
- 你现在最没把握的地方是什么

### 第三轮

- 有没有明确想保留或规避的竞品特征
- 有没有商业化或联机上的约束
- 这份文档最终是给谁看

## 输出要求

输出 `00_project_brief.md`，结构如下：

```md
# 项目简报

- 项目名称：
- 当前阶段：
- 日期：
- 信息可信度：

## 一句话定位

## 项目目标

## 目标平台与用户

## 核心卖点

## 多人联机价值

## 参考竞品

## 已确认信息

## 关键假设

## 待确认问题

## 建议下一步
```

## 完成标准

- 能让另一个 Agent 不看聊天记录，也知道这个项目大概是什么
- 用户当前缺失的信息被明确列出
- 可以无歧义地进入 `proposal-outline`
