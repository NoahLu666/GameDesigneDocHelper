# Proposal Draft

## 开场白

开始执行时，先向用户说明：

- 本 skill 用于公司内部立项正式成稿
- 默认项目属于微信小程序多人联机游戏平台
- 写作时会把“多人价值”当成核心判断项之一，不会把联机只写成附属功能
- 成稿完成后默认还会导出 `doc` 或 `docx`

## 目标

在信息不完全的前提下，产出一份结构完整、可讨论、可继续迭代的立项文档正式稿。

## 文档边界

默认输出的是立项文档，而不是完整量产策划案。

- 重点写清楚：为什么值得做、核心体验如何成立、MVP 如何验证
- 不要求把所有系统拆到实现级别
- 不要求写完整角色库、完整关卡库、完整数值表
- 只需要给出足以启动 MVP 研发的核心内容范围

## 必要输入

- `00_project_brief.md`
- `01_outline.md`
- 可选：`01_open_questions.md`
- 可选：`templates/proposal-template.md`
- 可选：`references/document-structure.md`

## 写作原则

1. 一级结构遵循标准模板。
2. 核心玩法、成长、联机等重点章节，优先使用“体验目标 -> 机制设计 -> 差异化 -> 风险”的写法。
3. 遇到不确定信息时：
   - 有高可信推断时，写成 `[假设]`
   - 明显缺信息时，写成 `[待确认]`
   - 需要后续测试时，写成 `[待验证]`
4. 不写空泛表达，例如“提升体验”“增强粘性”，除非后面跟具体机制或结果。
5. 尽量保留旧稿里“痛点 -> 对策”的表达方式。
6. 联机设计必须回答“多人为什么更好玩”，而不是只回答“多人怎么接入”。

## 强制包含内容

- 一句话定位
- 核心循环
- 核心乐趣点
- 与竞品的关键差异
- 多人联机价值
- 至少 3 条核心风险
- MVP 范围
- 下一步验证动作

## MVP 写作要求

- 如果原型内容很多，优先截取 1-2 个代表性关卡用于验证
- 角色只写 MVP 必需的少量代表性角色
- 数值只写原则和参考口径，不必写全量配置

## 输出要求

输出 `02_proposal.md`。

完成 Markdown 正式稿后，还必须追加 1 个导出动作：

- 默认导出 `项目名-立项文档.docx`
- 如果环境限制无法导出 `docx`，则导出 `doc`
- 导出成功后，检查文件是否能正常落盘

文末追加以下两节：

```md
## 附加说明

### 当前文档中的假设

### 当前文档中的待确认项

### 当前文档中的待验证项
```

## 自检清单

成稿前必须自检：

- 是否能让不了解项目的人读懂核心体验
- 是否解释了为什么它和竞品不同
- 是否说明了为什么值得立项
- 是否明确哪些是结论、哪些只是推断
- 是否给出了可执行的验证计划

## 完成标准

- 文档完整
- 风格统一
- 不确定项显式可见
- 可以直接进入讨论或评审
- 已生成对应的 `doc` 或 `docx` 版本
