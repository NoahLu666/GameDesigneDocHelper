# Proposal Retro

## 开场白

开始执行时，先向用户说明：

- 本 skill 用于公司内部立项文档的复盘和 workflow 优化
- 默认场景是微信小程序多人联机游戏平台
- 本阶段重点检查文档质量，也检查“多人联机”是否真正融入策划核心
- 还会检查最终交付格式是否已经稳定输出为 `doc` 或 `docx`

## 目标

在策划案完成后，输出面向下一次工作的复盘与流程优化建议。

## 输入

- `00_project_brief.md`
- `01_outline.md`
- `01_open_questions.md`
- `02_proposal.md`

## 检查维度

### 文档质量

- 哪些章节写得扎实
- 哪些章节明显偏空
- 哪些结论缺少论据
- 哪些联机设计只停留在表面

### 信息收集质量

- 哪些关键问题问晚了
- 哪些问题本不该问那么细
- 哪些地方可以用更高效的提问方式

### 工作流质量

- 哪一步最顺
- 哪一步最卡
- 哪一步最容易让 Agent 脑补
- 哪一步最值得模板化
- 哪一步最容易把多人玩法写空
- 最终导出 `doc/docx` 是否稳定

## 输出要求

输出两份文档。

### `03_retro.md`

```md
# 本次策划案复盘

## 做得好的部分

## 写得不够扎实的部分

## 应该更早确认的问题

## 仍然存在的关键风险

## 如果进入下一轮迭代，优先改哪里
```

### `03_workflow_improvements.md`

```md
# Workflow 优化建议

## 需要新增的问题

## 需要收紧的输出约束

## 可以模板化的段落

## 建议加入的检查清单

## 下一版 workflow 草案
```

## 输出风格

- 直说问题，不兜圈子
- 不把“用户没提供信息”简单归咎为用户问题
- 改进建议要能直接执行

## 必查项

- 是否过早把立项文档写成完整策划案
- 是否在用户给出竞品后仍未先搜索
- 是否缺少进度提示或 `A / B / C` 选项
- 是否遗漏最终 `doc/docx` 导出

## 完成标准

- 不是泛泛总结，而是明确指出下一版怎么改
- 复盘结论能反哺下一次 Intake / Outline / Draft
