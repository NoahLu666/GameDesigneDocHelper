# 文档结构参考

这份参考文档来自两类输入：

- 一份标准立项模板
- 一份实际写过的弹珠玩法立项文档

目标不是逐字复刻，而是提炼可复用结构。

## 一、模板文档的主结构

标准模板包含以下章节：

1. 文档基础信息
2. 项目概述
3. 市场与竞品分析
4. 题材与用户定位
5. 核心玩法设计
6. 外围成长设计
7. 多人联机与社交设计
8. 商业化构思
9. 风险与验证计划
10. 结论与立项建议

模板优点：

- 结构完整，适合统一输出
- 对立项判断和验证计划有保留位置
- 适合多人协作补写

模板缺点：

- 容易写成空话
- 缺少“玩家体验痛点 -> 对策”的推进逻辑
- 对玩法系统、数值来源、目标循环的表达约束偏弱

## 二、旧立项文档的有效写法

旧文档更像“真实策划推演”，有几类很有价值的表达方式：

### 1. 先定义核心体验

- 局内做什么
- 玩家爽点在哪里
- 与参考产品相比做了哪些改造

### 2. 明确写出痛点与解决方式

示例结构：

- `Build 不足 -> 加无尽模式 / 联机交换`
- `重复枯燥 -> 提速前期 / 加随机事件`
- `策略维度缺失 -> 给玩家更多生存空间`
- `经营建造简化 -> 改成神器系统`

这类写法非常适合保留到新 workflow 中。

### 3. 从局内到局外逐层展开

常见顺序：

1. 房间流程
2. 战斗规则
3. 关卡
4. 弹珠系统
5. 强化系统
6. 角色
7. 怪物
8. 随机事件
9. 结算
10. 局外养成
11. 商业化

### 4. 很适合形成“可配置点清单”

旧文档里有大量适合后续交给数值、程序、战斗策划继续展开的点，例如：

- 怪物刷新频率
- boss 出现条件
- 掉落数量
- 随机事件触发频率
- 神器等级与星级关系

## 三、推荐融合方式

建议把“模板的完整性”和“旧稿的推演感”结合起来：

- 一级章节仍然沿用标准模板
- 每个重点章节内部采用“体验目标 -> 机制设计 -> 差异化 -> 风险”的写法
- 对核心玩法、成长、联机这几章，优先写真实循环和取舍

## 四、推荐章节写法

### 项目概述

写法重点：

- 一句话定位
- 玩家在 10 秒内能理解的核心体验
- 为什么值得做

### 市场与竞品分析

写法重点：

- 市场机会不是泛泛而谈，而是要说明“谁被满足得不够好”
- 竞品结论最好写成“值得继承的点 / 应该规避的点”

### 核心玩法设计

写法重点：

- 单局循环
- 玩家操作
- 即时反馈
- build 形成方式
- 与竞品的关键改动

### 外围成长设计

写法重点：

- 局内资源如何带回局外
- 局外成长如何反馈局内
- 避免纯数值堆叠

### 联机与社交设计

写法重点：

- 联机不是“能一起玩”就够了
- 要说明联机增加了什么新决策、新乐趣、新传播点

### 风险与验证计划

写法重点：

- 核心风险必须可测试
- MVP 范围要能支持验证，而不是“先做大而全”

## 五、推荐输出语气

- 不写假大空口号
- 少写“提升体验”，多写“具体改善了什么体验”
- 对未确认项做显式标签，而不是躲过去
- 允许保留不确定性，但要给出建议方向
