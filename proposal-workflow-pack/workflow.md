# 策划案工作流

## 适用场景

- 公司内部游戏立项文档
- 微信小程序多人联机游戏策划案
- 多人游戏选题调研
- 简版立项 PRD
- 汇报型项目方案

## 固定前提

这套 workflow 默认服务于公司内部固定业务场景。

- 平台固定为微信小程序
- 产品固定为多人联机游戏平台
- 所有项目默认都需要支持多人游玩
- 输出对象默认是内部评审、制作人、主策或相关决策人

如果用户没有特别说明，Agent 应基于以上前提继续推进，不要反复追问这些基础条件。

## 立项文档边界

这套 workflow 的主要目标是产出立项文档，而不是完整量产策划案。

- 重点是说明为什么值得做、核心体验是什么、MVP 怎么验证
- 不要求一次写完完整系统细节
- 不要求补齐完整数值表
- 不要求设计完整章节数量、完整角色数量、完整长期运营内容
- 立项通过后，项目才进入 MVP 研发阶段

因此在 Draft 阶段，默认只写到“足以支撑立项判断和 MVP 启动”的深度。

## 最终交付格式

工作流中的分析与成稿过程可以使用 Markdown，但最终交付物默认必须包含：

- 1 份飞书友好的 Markdown 立项文档
- 保留对应 Markdown 版本，便于继续修改和二次导入

推荐使用方式：

- 在飞书云文档中直接上传 `.md` 文件
- 选择“创建飞书云文档”

## 总控模式

当前推荐将整套流程作为一个总控 skill 使用，名称为“立项助手”。

该 skill 默认负责：

- 自我介绍
- 使用说明
- Intake
- Outline
- Draft
- Export

同时支持两个手动触发分支：

- `/老板评审`
- `/立项复盘`

## Agent 执行原则

1. 先收集，再判断，再写作
2. 一次只推进一个阶段
3. 每个阶段必须有明确产物
4. 发现信息缺口时，优先一次只追问 1 个最关键问题
5. 无法确认的信息必须标注，不得脑补成结论
6. 复盘必须落到“下次怎么做更好”，而不是只做总结
7. 默认把“多人联机价值”视为立项核心判断项之一
8. 用户明确给出竞品时，先做基础检索，再继续提问
9. 选择题尽量使用 `A / B / C` 选项，方便快速回复
10. 每轮问答给出粗粒度进度提示，降低等待焦虑

## 立项入口分类

Intake 阶段优先判断项目属于哪一类，再决定后续问法。

### 1. 有多人联机原型游戏

重点不是重复发明玩法，而是判断：

- 原型游戏本身是否已经验证“好玩”
- 市场和数据表现是否支持参考
- 是否已有官方移动端版本
- 如果已有移动端版本，本项目还要不要做，差异化在哪里
- 放到公司平台后，竞争力和立项价值是什么
- 在免费游玩、IAP 持续运营前提下，商业化和留存如何成立

### 2. 有单机原型游戏

除竞品调研和商业化外，还要重点判断：

- 改成多人联机后，原作核心体验会不会被破坏
- 多人联机到底是自然增强，还是强行嫁接
- 多人玩法应该如何组织：协作、分工、对抗，还是混合
- 多人改造是否会改变节奏、目标与反馈结构

### 3. 没有成熟原型，或只有零散参考的创新项目

需要综合考虑：

- 玩法本身是否足够吸引人
- 多人联机价值是否天然成立
- 市场上是否已有相近验证对象
- 商业化和长期运营是否能闭环
- MVP 应如何验证核心乐趣

## 类型化追问原则

Agent 在 Intake 和 Outline 阶段，需要根据游戏类型主动切换追问重点，而不是使用统一问卷。

### 动作 / 射击 / 肉鸽 / 闯关类

优先细化：

- 基础操作方式
- 地图大小与空间结构
- 敌人刷新方式
- 边界规则与失败条件
- 角色差异
- 商店 / build / 强化结构
- 核心爽点和操作压力

### 模拟经营 / 资源循环类

优先细化：

- 核心双循环或多循环
- 生产、消耗与货币关系
- 用户的长期目标
- 经营压力与节奏
- 数值消耗点与回收点
- 题材包装和故事驱动

### 卡牌 / 策略 / 回合制类

优先细化：

- 核心决策点
- 信息呈现方式
- 构筑或养成结构
- 关卡目标与失败条件
- 节奏长短与思考负担

不同类型的游戏，在立项文档中需要展开的重点不同。商业化系统可以在立项阶段保留方向性描述，但核心玩法、操作、乐趣和验证方式必须写清楚。

## 阶段说明

### 阶段一：Intake

Agent 需要完成：

- 判断用户当前处于“想法期 / 调研期 / 成稿期”的哪一阶段
- 提炼一句话定位
- 明确项目目标、平台、受众、核心卖点
- 明确多人联机在该项目中承担什么价值
- 识别用户已经具备的信息和缺失信息
- 如用户已给竞品，先补齐竞品基础玩法与公开验证信息

阶段完成标准：

- 能产出一页项目简报
- 风险、假设、未知项已显式列出

### 阶段二：Outline

Agent 需要完成：

- 把简报映射到标准模板章节
- 判断哪些章节已有足够素材
- 判断哪些章节需要继续追问
- 给出章节写作优先级
- 检查“多人”是否只是人数包装，还是确实带来新体验

阶段完成标准：

- 已形成完整目录
- 每章都附带“已有信息 / 需要补充 / 建议写法”

### 阶段三：Draft

Agent 需要完成：

- 按模板产出立项文档正式稿
- 保留旧稿那种“体验痛点 -> 解决方式”的表达优势
- 避免空泛口号，优先写机制、目标、验证逻辑
- 对争议点给出建议版本，而不是模糊描述
- 对联机目标、协作或对抗关系、社交传播点给出明确说法

阶段完成标准：

- 形成可阅读、可讨论、可继续迭代的完整文稿
- 结尾附上 MVP 与验证计划

### 阶段四：Retro

Agent 需要完成：

- 检查哪些段落论据不足
- 检查哪些问题本应更早问
- 总结本次 workflow 哪些步骤顺畅、哪些步骤拖慢效率
- 生成下一次可复用的改进建议
- 回看“多人联机”是否真正融入了核心玩法与商业判断

阶段完成标准：

- 形成复盘文档
- 形成下一版 workflow 优化清单

### 阶段五：Export

Agent 需要完成：

- 将最终立项文档整理为飞书友好的 `.md`
- 默认命名为 `项目名-立项文档-飞书版.md`
- 检查导出文件是否成功落盘

阶段完成标准：

- 用户可以直接将 `.md` 上传到飞书并创建云文档
- Markdown 和最终导出文档内容保持一致

## 标准状态

- `NEEDS_INPUT`：缺少继续推进所需的关键信息
- `READY_FOR_NEXT_STEP`：本阶段产物已完成，可进入下一阶段
- `DRAFT_READY`：完整策划案已形成
- `RETRO_DONE`：复盘完成，改进项已沉淀

## 建议输出文件名

- `00_project_brief.md`
- `01_outline.md`
- `01_open_questions.md`
- `02_proposal.md`
- `03_retro.md`
- `03_workflow_improvements.md`
- `项目名-立项文档-飞书版.md`
